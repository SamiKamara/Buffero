2.  Product  Goals  
The  app  should  ship  with  these  goals:  
1.  Start  with  Windows  and  sit  in  the  tray.  2.  Detect  configured  games  automatically.  3.  Start  and  stop  replay  buffering  automatically  based  on  game  presence  or  focus.  4.  Save  the  last  15–120  seconds  on  a  global  hotkey.  5.  Produce  normal  MP4  clips  with  predictable  filenames.  6.  Keep  the  UI  minimal:  tray  menu,  settings  window,  and  simple  notifications.  7.  Avoid  vendor-specific  overlays  or  dependencies  on  NVIDIA/AMD  app  ecosystems.  
3.  Non-Goals  
These  are  intentionally  out  of  scope  for  the  first  version:  
●  live  streaming  ●  clip  editor  /  trimming  UI  ●  social  upload  /  sharing  ●  timeline  browser  ●  account  system  ●  WebView-based  frontend

●  plugin  ecosystem  ●  cross-platform  support

6.  Core  Design  Decision:  Segment-Based  Replay  Buffer  
The  replay  buffer  should  be  implemented  as  a  rolling  sequence  of  short  temporary  
segments
,
 
not
 
as
 
one
 
giant
 
endless
 
file
 
and
 
not
 
as
 
raw
 
uncompressed
 
frames
 
in
 
memory.
 
The
 
working
 
model
 
is:
 
●  record  into  short  temp  segments,  for  example  1–2  seconds  each  ●  keep  only  the  newest  segments  whose  combined  duration  is  within  the  configured  
buffer
 
size
 ●  delete  the  oldest  segment  as  each  new  segment  is  finalized  ●  when  the  user  hits  the  hotkey,  snapshot  the  segment  list  that  covers  the  requested  
replay
 
window
 ●  export  those  segments  into  a  final  MP4  clip  in  the  background  
This  is  the  key  design  choice  for  keeping  memory  usage  sane  and  making  crash  recovery  
manageable.
 
It
 
also
 
keeps
 
the
 
app
 
closer
 
to
 
a
 
DVR
 
model
 
than
 
a
 
general-purpose
 
recorder.
 
For  the  first  version,  the  segment  duration  should  be  fixed  globally,  probably  2  seconds .  
That
 
gives
 
reasonable
 
precision
 
without
 
creating
 
too
 
many
 
files.
 
7.  Capture  Model  
7.1  Target  selection  
The  app  should  support  two  capture  target  modes:  
●  Window  capture  ●  Display  capture  
Default  behavior  should  be  window  capture  for  configured  games.  Display  capture  can  be  a  
manual

Windows.Graphics.Capture  supports  acquiring  frames  from  a  display  or  an  application

7.2  Auto-start  logic  
Game  auto-start  should  be  based  on  a  profile  match  plus  a  focus  rule :

●  a  configured  executable  is  running  ●  that  executable  has  a  visible  top-level  window  ●  optionally,  that  window  is  the  current  foreground  window  
Foreground  changes  can  be  observed  with  SetWinEventHook(EVENT_SYSTEM_FOREGROUND,  ...),  which  is  more  responsive  than  
polling
 
alone,
 
but

should

Microsoft’s
 
docs

 SetWinEventHook requires  a  message  loop  on  the  calling  thread.  
Recommended  behavior:  
●  when  a  matching  game  appears,  move  to  Armed  ●  when  its  target  window  is  valid  and  capture  can  begin,  move  to  Capturing  ●  when  the  game  exits  or  loses  eligibility  for  a  sustained  period,  stop  capture  and

Use  a  debounce  of  around  2–3  seconds  to  avoid  flapping  during  launcher-to-game

8.  Video  Pipeline  
The  video  pipeline  should  be  built  around  Windows.Graphics.Capture  frame  acquisition  
plus
 
a
 
Media

.
 
Microsoft’s

Logical  flow:  
1.  Start  capture  session  for  target  window/display.  2.  Receive  frames  from  the  capture  API.  3.  Normalize  timing  and  frame  pacing.  4.  Hand  frames  to  encoder  path.  5.  Write  output  into  current  temp  segment.  6.  Rotate  segment  at  fixed  duration  boundary.  
Recommended  initial  output  profile:  
●  container:  MP4  ●  video  codec:  H.264  ●  frame  rate:  60  fps  default,  30  fps  fallback  ●  quality  preset:  user-selectable  simple  slider  ●  resolution:  native  by  default,  optional  scale-down  
The  MVP  should  prioritize  robustness  over  a  giant  matrix  of  encoding  options.  The  settings  
UI
 
should

10.  Export  Strategy  
When  the  user  presses  the  save  hotkey,  the  export  path  should:  
1.  Capture  the  current  replay-window  snapshot  from  the  segment  index.  2.  Mark  the  exact  segment  range  and  time  bounds.  3.  Queue  an  export  job.  4.  Continue  live  buffering  without  blocking  the  current  session.  5.  Produce  a  final  MP4  in  the  configured  output  directory.  
For  the  MVP,  export  correctness  matters  more  than  absolute  speed.  If  direct  remux  of  
segments

avoids

live
 
replay
 
path

17.  Reliability  and  Recovery  
The  app  should  expect  these  failures:  
●  target  window  closes  ●  display  mode  changes  ●  encoder  init  fails  ●  output  drive  is  full  ●  temp  folder  exceeds  budget  ●  audio  device  changes  ●  access  to  target  becomes  invalid  
Recovery  policy:  
●  retry  attach  for  short  transient  failures  ●  stop  and  restart  capture  session  on  device  changes  ●  refuse  export  when  disk  space  is  critically  low  ●  log  all  transitions  with  reasons  ●  keep  the  app  alive  even  if  one  capture  session  fails

21.  Bottom-Line  Recommendation  
Build  the  first  version  as  a  single-process  native  tray  app  with:  
●  WPF  for  UI  ●  RegisterHotKey  +  WM_HOTKEY  for  save  hotkey  ●  SetWinEventHook  +  light  process  scan  for  game  detection  ●  Windows.Graphics.Capture  for  video  frames  ●  WASAPI  loopback  for  system  audio  ●  Media  Foundation  for  encoding/export  ●  segment-based  rolling  temp  storage  for  the  replay  buffer
